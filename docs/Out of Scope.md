# Out of Scope
This page tracks work that is out-of-scope, but still worth documenting.

- Running more than one Incus cluster member (0.x initializes Incus as a single-member cluster from the start — see `Decisions.md` § App Manager HA; scaling to N members, and the operator workflow for joining them, is deferred). Revisit wrapping Operations Center once real multi-member clustering is on the table.
- GitOps auto-apply and rollback (today: diff-and-warn only)
- Private repos, repo-sharing across environments
- IPv6, DHCP/DNS write-back
- Cert rotation/revocation, moving off self-signed to a real CA
- Multi-disk / multi-NIC instance overrides
- Commit-hash-per-node tracking
- Phone-home / hardware-manifest reporting over Tailscale (a one-shot manifest ping from a node to the app — distinct from the continuous logs/metrics telemetry now in Phase 3 scope, see `Roadmap.md`/`Decisions.md` § Metrics — and distinct from the pre-install helper-OS registration/confirmation flow under `Decisions.md` § Multi-OS support, which is interactive/blocking rather than a one-shot post-install ping)
- Migrating the web app's own runtime from its Docker Compose/binary deployment into the IncusOS fleet
- Running the web container as a non-root user (defense-in-depth): switch the final Dockerfile stage to distroless's `nonroot` user (uid 65532) / the `:nonroot` image tag. Tackle after Phase 3 — 0.x is single-user/no-auth (below), so root-in-container isn't a headline risk yet. Read-only config mounts (e.g. `CLIENT_CERT_PATH`) and `/tmp` (mode 1777) already work as non-root; the one thing to verify first is that a mounted `STORE_PATH` volume (the sqlite file) is writable by uid 65532, or the store fails to open on startup.
- Standardized schema output for the fleet-definition format and/or the web app's REST API: JSON Schema generated from `internal/config`'s structs (editor autocomplete, CI validation of fleet YAML) and/or OpenAPI for the existing HTTP API (see `Architecture.md` § HTTP API). No GraphQL — the app's scope is simple CRUD-ish config management, not query-heavy enough to justify a schema-first GraphQL layer. This is the expected growth path for validation if it ever outgrows the hand-rolled `config.Validate` chosen in `Decisions.md` § Validation approach — generate the schema *from* the (typed `net/netip`) structs, rather than adopt a schema-DSL library.
- Multi-user on web app. 0.x should focus on just 1 user for now. No auth needed.
- Supporting more than IncusOS as a managed-node target. Two directions are planned but not yet built (see `Decisions.md` § Multi-OS support): plain Debian w/ Incus, installed via a systemd-repart-based helper OS with a pre-install hardware-registration/confirmation flow; and Talos (Kubernetes nodes), single-node install first, full cluster lifecycle (control-plane/worker roles, `talosctl bootstrap`) deferred further still. Both are blocked on the OS-target abstraction and node→app networking work described there.
- DB-specific (or other renderer-specific) write-promotion wiring for `kind: App` beyond the generic `Renderer.Promote` seam (see #92, `docs/AppManager.md`, and `Decisions.md` § Stateful app support for the shape this would take)
- Backup/restore for `kind: App` data/volumes. Note this is a prerequisite for any real managed database, not a nice-to-have: replication is not backup — a `DROP TABLE` replicates perfectly (`Decisions.md` § Stateful app support).
- **Multi-instance renderers** — the *cardinality* half of this is in scope (`replicas: N` / `per-node` on `kind: App`, and a reconcile loop keyed by `(App, member)`; see `docs/AppManager.md`), but the renderers that would need N > 1 are not: a database (class 3) or a quorum member (class 5) needs volumes, version-skew classification and DB-native HA before `replicas: 3` means anything. See `docs/AppClasses.md` and `Decisions.md` § Stateful app support.
- **Cross-node App placement/scheduling/bin-packing.** `kind: App` has no placement field in 0.x — Incus's own scheduler decides, which is moot on today's single-member cluster. Placement is *not* `replicas`: that says only *how many*, never *where* (`App.Node` was deleted precisely because it meant both). When real placement is eventually needed, the intended mechanism is Incus's own cluster groups (tag members with a capability, target creation at the group) via a **separate** field, invalid in combination with `replicas: per-node`, not an operator naming individual nodes — see `Decisions.md` § App Manager HA and `docs/AppClasses.md` § Placement is a third axis.
- Persistent volumes / storage schema for `kind: App`. There's no volume field at all today, which is what confines every App to being stateless (or destructively updated — `docs/AppClasses.md` class 6). Needed before a database, and it brings its own lifecycle questions (does a volume outlive its generation? per-member or per-generation? orphan policy?).
- Version-skew classification for `kind: App` upgrades. A bump is detected as "declared `image` differs from the live instance's tag" — deliberately coarse, and sufficient for every renderer 0.x ships. It is not sufficient for classes 3/5: Postgres physical replication requires matching major versions (16 → 17 can't blue-green at all — that's `pg_upgrade`/logical replication), and Kubernetes has an N-2 skew policy plus control-plane-first ordering. See `docs/AppClasses.md` § Version skew.
- Run-to-completion / batch Apps (a migration or backup job). `Renderer.Healthy` answers "healthy right now"; a job wants "exited 0", and blue-green is meaningless for it. This is the natural home for a database backup sidecar if one is ever needed.
- A stable endpoint/service address for `kind: App`. Apps get whatever address Incus gives them; there's no field for a static IP or a service name, and DNS write-back is out of scope (above). This is unresolved rather than decided — a proxy fronting a database needs a stable address to be useful, which would make it `docs/AppClasses.md` class 6 (scarce resource, needing a floating-IP handoff) rather than the class 1 it otherwise looks like.
- App-level dependency ordering (no "App B waits on App A" graph)
- Automatic rollback after a successful App promotion (health-gating is strictly pre-promotion)
- Orphaned-App teardown (App removed from git; live instances not auto-deleted)
- Operator-facing UI/API for App/rollout status (beyond `incus list`)
- Self-hosted ACME CA (e.g. [step-ca](https://smallstep.com/docs/step-ca/)) so nodes fetch a real, chain-verifiable server certificate for their Incus API instead of the self-signed one Incus generates at first boot. Raised alongside #91: `internal/nodeprovision`/`cmd/validate-91-harness`'s direct-to-Incus TLS clients currently set `InsecureSkipVerify: true` (dismissed as an accepted CodeQL finding, not fixed), since there's no CA to chain-verify against and nothing in the seed tells the web app what a node's server cert will be in advance — trust instead comes from the WireGuard tunnel's own peer authentication plus the client cert. A self-hosted ACME CA the web app trusts, with IncusOS configured (seed-time or post-boot) to fetch its server cert via ACME from that CA, would let these calls do real server-certificate verification instead of skipping it — closing that gap properly rather than just accepting it. Overlaps with the existing "moving off self-signed to a real CA" item above, but concrete enough (specific tool, specific trigger) to track on its own. Needs: confirming IncusOS actually exposes an ACME-configuration hook (seed file or `incus-osd` API, same kind of gap the Tailscale seed-hook investigation in `Decisions.md` §8 hit), and deciding where step-ca itself would run (web app process vs. a separate managed service).
